# CLAUDE.md — 범용 작업 지침

이 문서는 프로젝트에 무관하게 적용되는 **범용 작업 지침**이다. 모든 Claude 에이전트는 이 지침을 준수한다.
프로젝트 특화 정보(구조·명령어·설계 내용)는 이 문서에 두지 않으며, 아래 Entry-Point 규약을 통해 얻는다.

## Entry-Point 규약 — 작업 진입점

**모든 작업은 `docs/Entry-Point.md`를 읽는 것으로 시작한다.**

- `docs/Entry-Point.md`는 프로젝트의 설계 문서 체계를 안내하는 진입점 문서다. 에이전트는 이 문서의 매핑에 따라 **작업 속성에 맞는 설계 문서를 선택해 읽고, 그 내용에 근거해서만 작업한다.**
- 프로젝트 개요, 실행·테스트 명령, 아키텍처, API 명세, 테스트 프로세스 등 프로젝트 특화 정보는 전부 Entry-Point.md가 안내하는 문서에서 얻는다. 학습 데이터나 일반적 관례에 의존해 추측하지 않는다.
- 서브에이전트에 작업을 위임할 때도 Entry-Point.md 경유 규약(또는 필요한 설계 문서의 경로)을 프롬프트에 명시한다.
- `docs/Entry-Point.md`가 없는 프로젝트에서는 작업 전에 사용자에게 알리고 작성을 제안한다.

## 핵심 개발 원칙 — 설계가 코드에 선행한다

모든 변경은 다음 순서를 따른다.

1. **철저한 설계 합의** — 사용자와 설계 수준에서 합의가 이루어지기 전에는 코드를 작성하지 않는다.
2. **구조적 문서화** — 합의된 설계를 문서로 먼저 기록한다.
3. **문서 근거 구현** — 코드는 문서화된 내용에 철저히 근거해서만 작성한다. 문서에 없는 동작을 임의로 추가하지 않는다.

문서와 코드가 불일치하는 상황을 발견하면 구현을 계속하지 말고 사용자에게 보고하여 문서를 먼저 정정한다.

## 작업 프로세스 — PDCA

모든 코드 변경 작업(신규 기능, 수정, 리팩토링)은 PDCA 사이클을 따른다.
목적은 **한 번에 기존 코드 여러 개가 동시다발적으로 수정되는 상황을 방지**하고, 모든 변경이 사용자 승인을 거치도록 보장하는 것이다.

### Plan — 사용자의 작업 계획 질의

- 사용자가 작업 계획이나 요구사항을 입력하면 사이클이 시작된다.
- 에이전트는 요구사항이 모호하면 이 단계에서 질문하여 해소한다. 이 단계에서는 코드를 수정하지 않는다.

### Do — 변경 계획 문서 작성 (`/docs/tmp`)

- 에이전트는 작업 계획을 토대로 코드베이스를 분석하여 변경점·추가 사항을 파악하고, 그 결과를 `docs/tmp/` 하위에 변경 계획 문서로 기록한다. **이 단계에서 기존 코드를 직접 수정하지 않는다.**
- 문서 파일명: `docs/tmp/YYYY-MM-DD-{작업-주제}.md`
- 문서 상세 수준은 **파일별 변경 명세**로 한다:
  - 변경 대상 파일의 전체 목록 (신규 / 수정 / 삭제 구분)
  - 각 파일의 변경 내용 서술
  - 핵심 코드(시그니처, 주요 로직)만 스니펫으로 제시 — 전체 코드를 문서에 나열하지 않는다
- **기존 설계 문서 수정안을 반드시 포함한다.** 변경이 프로젝트 설계 문서에 영향을 주면, 해당 문서의 수정안(섹션·버전 표기 포함)을 tmp 문서에 함께 기록한다. 문서-코드 정합성은 항상 유지되어야 한다.
- 테스트가 필요한 변경이면 프로젝트의 테스트 프로세스 문서(Entry-Point.md에서 확인)에 따라 테스트 케이스 도출안도 tmp 문서에 포함한다.

### Check — 사용자 검토 및 승인

- 사용자가 `docs/tmp/` 문서를 검토·검증하고 변경 사항을 승인한다.
- 에이전트는 승인 없이 Act로 진행하지 않는다. 사용자가 수정을 요구하면 tmp 문서를 갱신하고 다시 Check를 받는다.

### Act — 반영 및 정리

반영 주체는 **작업 규모에 따라 유동적으로 결정**한다 (사용자가 지정).

- **에이전트가 반영하는 경우**: tmp 문서에 기록된 내용대로만 코드와 설계 문서를 코드베이스에 반영한다. 문서에 없는 변경이 필요해지면 중단하고 사용자에게 보고한다.
- **사용자가 직접 반영하는 경우**: 사용자가 "직접 반영했다"고 알리면, 에이전트는 tmp 문서의 내용이 실제 코드베이스에 어떻게 반영되었는지 **Cross-Check**를 수행한다. 누락·불일치가 있으면 삭제 전에 보고한다.
- 반영 완료 후 에이전트는:
  1. 관련 테스트를 실행하여 결과를 보고한다 (테스트 명령은 Entry-Point.md 참조 — 실패 시 원문 출력과 함께 보고).
  2. 검증이 끝나면 해당 tmp 문서를 삭제한다.
  3. 커밋 컨벤션에 따라 커밋을 생성한다.

## 커밋 컨벤션

- 형식: `{타입}: {한국어 제목}` — 타입은 `Feat`, `Fix`, `Refactor`, `Test`, `Docs`
- 복합 작업은 타입과 제목을 각각 `/`로 병기한다. 예: `Feat / Docs: 회원 탈퇴 백엔드 API 구현 / 개발 문서 수정`
- 코드 변경과 그에 따른 설계 문서 갱신은 같은 커밋에 담는다.

## 에이전트·모델 운용 지침

토큰 소모를 효율화하면서 품질을 유지하기 위해, 작업 특성에 따라 모델을 구분해 사용한다.
메인 에이전트는 PDCA의 **Do 단계**에서 아래 기준으로 서브에이전트를 생성해 작업을 위임할 수 있다.

### 모델별 적합 작업

| 모델       | 적합한 작업                                                                                                                             | 예시                                                                           |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **Opus**   | 설계 판단이 필요한 작업. 아키텍처 결정, 트레이드오프 분석, 변경 계획 문서(tmp) 작성, Cross-Check 검증, 정합성이 중요한 도메인 로직      | 동시성·트랜잭션·캐시 무효화 로직 설계, 스키마 확장 설계, PDCA tmp 문서 작성    |
| **Sonnet** | 명세가 확정된 표준 구현. 설계 문서·tmp 문서에 근거한 코드 작성, 테스트 코드 구현, API 엔드포인트 추가, UI 컴포넌트 구현, 설계 문서 반영 | Act 단계 코드 반영, 확정된 테스트 케이스 구현, CRUD 구현, 페이지·컴포넌트 구현 |
| **Haiku**  | 판단이 거의 필요 없는 대량·반복 작업. 코드베이스 탐색·검색, 파일 목록화, 기계적 치환(rename, import 정리), 포맷팅, 문서 요약            | Do 단계 사전 조사(영향 범위 파악을 위한 탐색), 커밋 이력 요약, 단순 lint 수정  |

### 운용 규칙

- **역할 배분의 원칙**: 판단·설계는 상위 모델이, 확정된 명세의 실행은 하위 모델이 맡는다. 명세가 모호한 작업을 하위 모델에 위임하지 않는다.
- Do 단계에서 코드베이스 영향 범위 파악이 넓은 경우, 탐색은 Haiku/Explore 서브에이전트에 위임하고 메인 에이전트는 결론만 받아 tmp 문서를 작성한다.
- Act 단계에서 tmp 문서가 충분히 상세하면(파일별 명세 + 스니펫) 반영 작업을 Sonnet 서브에이전트에 위임할 수 있다. 이때 tmp 문서 경로와 준수 의무("문서에 없는 변경 금지")를 프롬프트에 명시한다.
- 서브에이전트 산출물은 메인 에이전트가 tmp 문서(또는 설계 문서)와 대조 검증한 후 사용자에게 보고한다.
- 단일 파일 수준의 소규모 작업은 서브에이전트 없이 메인 에이전트가 직접 처리한다 — 위임 오버헤드가 절약보다 크다.

## 언어

- 사용자와의 대화, 설계 문서, tmp 문서, 커밋 메시지는 **한국어**로 작성한다.
- 코드 식별자·API 응답 필드는 영어를 유지한다.
