설계 문서는 AI 에이전트를 더 효율적으로 만드는가

동일한 스텁 구현 과제를 두 조건 모두 TDD로 개발하되, 한쪽만 설계 문서가 보조하도록 두고(C1=문서 有, C2=문서 無) 완료까지의 비용·시간·토큰을 측정했다. 독립 변인은 설계 문서의 존재 하나다. 핵심 질문: 설계 문서가 TDD를 보조하면 더 효율적인가?

한눈에 — 가설 판정
H1 · 문서가 더 저렴한가 방향 일치, 통계적으로는 미확정

복잡한 코드에서만, 그것도 완만하게

동시성·실패복구가 얽힌 T-chem에서 문서 有(C1)가 모든 지표에서 저렴·빠름(배수 전부 1.0의 오른쪽). 그러나 정리된 n=10에서 유의에 도달한 지표는 없다 — 비용 1.09×(p=.166), 벽시계 1.13×(p=.130), 출력 토큰 1.10×(p=.104). 로컬 도구시간만 marginal(p=.089).

H2 · 효과는 코드 성격에 의존하는가 강하게 지지됨

설계 의도가 코드 밖에 있을 때만 작동

순수 도메인 로직 T-score에서는 문서 효과가 사실상 0(모든 지표 ≈1.0×). 테스트+프롬프트만으로 완전히 결정되는 코드엔 문서가 더 줄 것이 없다. 두 대상의 대비가 이 실험의 가장 견고한 발견이다.

종합하면, “설계 문서가 TDD를 보조하면 더 효율적인가”의 답은 방향으로는 그렇다, 다만 통계적으로는 미확정이다 — 설계 의도가 코드·테스트 밖에 흩어진 코드(T-chem)에서만 문서 有가 모든 지표에서 저렴·빠르며(중앙/평균 기준 약 7–13%), 그 방향은 일관되나 조건당 n=10에서 유의에는 이르지 못한다. 단순·자기완결 로직에서는 그 이점조차 없다.

효과 크기 개관

C2 ÷ C1 배수 — 1.0에서 오른쪽으로 갈수록 “문서가 아꼈다”

각 지표의 효과 크기 비(문서 無 ÷ 문서 有) — 검정에 정렬: 정규 지표는 평균비, 비정규 지표는 중앙비. 1.0 세로선이 “차이 없음”. 막대가 오른쪽으로 뻗을수록 문서 없는 C2가 그만큼 더 썼다는 뜻. 채운 점은 통계적으로 유의(p<.1), 빈 점은 유의하지 않음.

T-chem · 동시성·캐시·실패복구

C1 n=10 · C2 n=10 · Shapiro→t/U

T-score · 순수 도메인 로직 (대조군)

C1 n=10 · C2 n=10 · Shapiro→t/U

T-chem의 막대들은 일제히 1.0의 오른쪽에 서고, T-score의 막대들은 1.0에 달라붙는다. 이 시각적 대비가 곧 H2다.

분포 · T-chem

복잡한 코드 — 겹치지만 일관되게 밀린다

런 하나가 점 하나. 상자는 IQR(25–75%), 굵은 세로선이 중앙값. C1(티일)과 C2(앰버)의 상자가 상당히 겹치는 것이 보이는데, 이 겹침이 곧 “방향은 맞지만 p값이 경계선”인 이유다.

C1 · 문서 有 C2 · 문서 無 (TDD) 각 점 = 1 런
분포 · T-score (대조군)

단순한 코드 — 두 조건이 포개진다

여기서는 C1과 C2 상자가 거의 완전히 겹친다. 문서가 있으나 없으나 결과가 같다 — 테스트와 프롬프트만으로 이 로직이 온전히 결정되기 때문이다.

C1 · 문서 有 C2 · 문서 無 (TDD) 각 점 = 1 런
해석 시 주의할 두 가지

숫자를 잘못 읽지 않기

달러는 시간 노력을 과소평가한다

T-chem에서 시간 격차(벽시계 1.13×)가 비용 격차(1.09×)보다 다소 크다. C2가 더 하는 일의 상당수가 값싼 cache_read(문맥 되읽기)라 달러로는 덜 잡히기 때문 — 문서는 “하는 일의 양”을 줄이지만 그 절약분이 저렴한 문맥읽기라 달러 절감이 완만하다. 다만 정리된 n=10에선 두 격차 모두 유의하지 않아 방향 수준의 해석이다.

p값만 보면 안 되는 사례

T-score의 로컬 도구시간은 1.11×, p=.031로 “유의”하게 나온다 — 정리 후에도 T-chem·T-score 통틀어 p<.05에 도달한 유일한 지표다. 그러나 실제 차이는 8.4s → 9.3s, 단 1초다. 통계적 유의성과 실질적 크기는 별개 — gradle 한 번 더 돈 정도의 무의미한 차이이며, T-score의 “무효과” 결론을 뒤집지 않는다.

턴 수는 어느 셀에서도 유의하게 달라지지 않았다(배수 0.95–1.02×, 전부 ns). 문서는 작업 단계 수를 줄이지 않는다 — 각 단계에서 하는 일(문맥 읽기·코드 재작성·테스트 재실행)을 가볍게 만들 뿐이다.

방법론 · 통제 · 위협

어떻게 측정했나

  • 2 조건 × 2 대상. 두 조건 모두 원본 기준 테스트로 개발(TDD)하며 독립 변인은 설계 문서의 존재 하나 — C1=문서 有, C2=문서 물리 삭제. 대상은 ScoringService(단순)·ChemistryCacheService(복잡).
  • 격리. 매 런은 git archive로 만든 히스토리 제거 사본에서 수행 — 원본 구현·문서를 git show로도 복원 불가.
  • 스텁화. 대상 메서드 본문을 제거(UnsupportedOperationException), 주석 전량 삭제 후 프롬프트로 이관 → 코드 내 스펙 누설 0, 양 조건 동일 전달.
  • 성공 오라클. 기준 테스트 클래스만 실행, 종료코드로 green 판정. 정지 조건을 테스트에 위임해 변인을 문서 하나로 통제.
  • 통제. 모델 claude-sonnet-5 고정(보조 haiku <1%), 동일 프롬프트·동일 커밋·순차 실행. CLAUDE.md(PDCA)는 축 분리를 위해 모든 조건에서 제외.
  • 검정. 지표·군마다 Shapiro-Wilk 정규성 검정(α=.05) 후, 둘 다 정규면 Welch t(양측·등분산 미가정), 아니면 Mann–Whitney U(양측). 효과 크기는 검정에 정렬 — 정규=평균비, 비정규=중앙비.

타당성 위협 (함께 보고)

  • 주석→프롬프트 이관의 상한. C2도 제거된 주석을 그대로 받으므로, C1의 잔여 이득은 “문서에만 있는 정보”로 제한된다. 이는 보수적 설계 — 그럼에도 C1이 싸면 주장이 강해진다.
  • 대상 2종의 한계. 두 서비스로 H2를 완전히 일반화할 수 없다. “경향의 방향”으로만 사용.
  • 모델 한정. 결론은 Sonnet에 한정. 다른 모델로의 일반화는 미검증.
  • 표본·검정력. 조건당 n=10. T-chem에서 비용 ~9%·시간 ~13% 차이가 관측되나 소표본이라 유의에 이르지 못한다 — 방향은 일관하나 단정은 보류. 확정하려면 표본을 키워야 한다.

데이터 품질: 분석 대상은 정리된 40런(조건당 10)으로 전부 성공, 상한 초과·타임아웃 0건. 초기 테스트 용도로 수행한 예비 iteration은 노이즈이므로 분석에서 제외했다.
