설계 문서는 AI 에이전트를 더 효율적으로 만드는가

동일한 스텁 구현 과제를 두 조건 모두 TDD로 개발하되, 한쪽만 설계 문서가 보조하도록 두고(C1=문서 有, C2=문서 無) 완료까지의 비용·시간·토큰을 측정했다. 독립 변인은 설계 문서의 존재 하나다. 핵심 질문: 설계 문서가 TDD를 보조하면 더 효율적인가?

한눈에 — 가설 판정
H1 · 문서가 더 저렴한가 미지지 — 감지 가능한 효과 없음

표본을 키우자 방향이 사라졌다

H1이 가장 강하게 나타날 것으로 본 T-chem의 표본을 조건당 ≈30(C1 N=29 · C2 N=30)으로 확대하자, 모든 지표가 유의하지 않고 점추정이 동률로 수렴했다 — 비용 1.03×(p=.369), 벽시계 1.06×(p=.170). n=10의 방향 신호(약 7–13%)는 재현되지 않았고, 로컬 도구시간은 부호가 뒤집혔다(1.10×→0.98×).

H2 · 효과는 코드 성격에 의존하는가 약화 — 대비 소멸

두 대상이 모두 무효과로 수렴

n=10에서 H2를 지지하던 근거는 T-chem 효과 有 / T-score 無의 대비였다. 이제 T-chem마저 ≈null로 수렴해 두 대상이 모두 무효과가 되면서 그 대비가 사라졌다. 대조군 T-score는 v0.6과 동일하게 무효과(모든 지표 ≈1.0×).

종합하면, Sonnet·이 두 서비스·“테스트가 주어진 TDD” 기준선에서는 설계 문서의 물리적 존재가 완료까지의 비용·시간·턴을 통계적으로 감지될 만큼 줄이지 못한다. T-chem에서 n=10일 때 보이던 겉보기 이점은 표본을 키우자 사라졌으므로 소표본 노이즈였을 가능성이 크다. 이는 “효과가 정확히 0”이라는 증명이 아니라(검정력 한계·§방법론), 확대 표본이 문서 우위의 근거를 뒷받침하지 못한다는 뜻이다.

효과 크기 개관

C2 ÷ C1 배수 — 1.0에서 오른쪽으로 갈수록 “문서가 아꼈다”

각 지표의 효과 크기 비(문서 無 ÷ 문서 有) — 검정에 정렬: 정규 지표는 평균비, 비정규 지표는 중앙비. 1.0 세로선이 “차이 없음”. 막대가 오른쪽으로 뻗을수록 문서 없는 C2가 그만큼 더 썼다는 뜻. 채운 점은 통계적으로 유의(p<.1), 빈 점은 유의하지 않음.

T-chem · 동시성·캐시·실패복구

C1 N=29 · C2 N=30 · Shapiro→t/U

T-score · 순수 도메인 로직 (대조군)

C1 N=10 · C2 N=10 · Shapiro→t/U

N≈30으로 키운 T-chem의 막대들은 이제 1.0 세로선에 바짝 달라붙는다(전부 유의하지 않음, 로컬 도구시간은 왼쪽으로 넘어감). 대조군 T-score와 마찬가지로 “차이 없음”에 수렴한 것이며, v0.6에서 보이던 오른쪽 쏠림은 표본 확대 후 사라졌다.

복잡한 코드 — N≈30에서 두 조건이 포개진다

런 하나가 점 하나. 상자는 IQR(25–75%), 굵은 세로선이 중앙값. C1(티일)과 C2(앰버)의 상자가 거의 완전히 겹친다 — 표본을 조건당 ≈30으로 키우자 중앙값·평균이 사실상 포개지고, 모든 지표가 유의하지 않다.

왜 효과가 없는가 — 보수적 설계의 귀결

본 실험은 보수적으로 설계됐다 — (a) 제거한 주석 스펙을 양 조건 프롬프트에 동일 전달하고, (b) 정지 판정을 테스트 오라클에 위임했다. 즉 C2도 스펙과 테스트를 충분히 받으므로, C1이 추가로 얻는 “문서에만 있는 정보”가 Sonnet의 구현 경로를 유의하게 바꿀 만큼 크지 않았다고 읽힌다. 문서의 가치가 이 조건에서 발현되지 않은 것이지, 다른 모델·과제·기준선에서의 가치를 부정하는 결과는 아니다.

p값과 실질 크기는 별개 (대조군)

T-score의 로컬 도구시간은 1.11×, p=.031로 80런 통틀어 유일하게 p<.05에 도달한 지표다. 그러나 실제 차이는 8.4s → 9.3s, 단 1초다. 통계적 유의성과 실질적 크기는 별개 — gradle 한 번 더 돈 정도의 무의미한 차이이며, T-score의 “무효과” 결론을 뒤집지 않는다.

턴 수는 어느 셀에서도 유의하게 달라지지 않았다(배수 0.95–1.04×, 전부 ns). 문서는 작업 단계 수를 줄이지 않으며, N≈30에서는 단계당 작업량(비용·토큰)조차 유의하게 줄이지 못했다.

방법론 · 통제 · 위협

어떻게 측정했나

  • 2 조건 × 2 대상. 두 조건 모두 원본 기준 테스트로 개발(TDD)하며 독립 변인은 설계 문서의 존재 하나 — C1=문서 有, C2=문서 물리 삭제. 대상은 ScoringService(단순)·ChemistryCacheService(복잡).
  • 격리. 매 런은 git archive로 만든 히스토리 제거 사본에서 수행 — 원본 구현·문서를 git show로도 복원 불가.
  • 스텁화. 대상 메서드 본문을 제거(UnsupportedOperationException), 주석 전량 삭제 후 프롬프트로 이관 → 코드 내 스펙 누설 0, 양 조건 동일 전달.
  • 성공 오라클. 기준 테스트 클래스만 실행, 종료코드로 green 판정. 정지 조건을 테스트에 위임해 변인을 문서 하나로 통제.
  • 통제. 모델 claude-sonnet-5 고정(보조 haiku <1%), 동일 프롬프트·동일 커밋·순차 실행. CLAUDE.md(PDCA)는 축 분리를 위해 모든 조건에서 제외.
  • 검정. 지표·군마다 Shapiro-Wilk 정규성 검정(α=.05) 후, 둘 다 정규면 Welch t(양측·등분산 미가정), 아니면 Mann–Whitney U(양측). 효과 크기는 검정에 정렬 — 정규=평균비, 비정규=중앙비.

타당성 위협 (함께 보고)

  • 주석→프롬프트 이관의 상한. C2도 제거된 주석을 그대로 받으므로, C1의 잔여 이득은 “문서에만 있는 정보”로 제한된다(보수적 설계). 이번 결과는 T-chem에서도 그 잔여 이득이 감지 수준에 못 미침을 시사한다.
  • 대상 2종의 한계. 두 서비스로 결론을 일반화할 수 없다. “이 두 코드에서의 관측”으로만 사용.
  • 모델 한정. 결론은 Sonnet에 한정. 다른 모델로의 일반화는 미검증.
  • 검정력(핵심). T-chem 비용 효과는 d≈0.24(소효과)로 N≈30에서 p<.05로 잡을 검정력이 낮다 — 즉 “효과 0의 증명”이 아니라 “감지 가능한 효과 부재”다. 다만 점추정이 null로 이동하고 부호 일관성까지 잃은 점이 단순 p>.05보다 강한 근거이며, 최소한 v0.6의 “7–13% 비용 우위”는 확대 표본에서 지지되지 않는다.

데이터 품질: 분석 대상은 T-chem 59런(29+30)·T-score 20런으로 전부 성공, 상한 초과·타임아웃 0건. cli_error 1건(T-chem C1)은 지표 유실로 제외(N=29). 초기 테스트 용도 예비 iteration도 노이즈이므로 제외했다.
