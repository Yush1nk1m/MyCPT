## 1. 서비스 개요

### 1.1 서비스명

**MyCPT** (나의 역량, My ComPeTency)

### 1.2 서비스 개요

DISC 이론 기반의 24문항 강제선택 검사를 통해 나의 역량을 분석하고, 동료와의 협업 케미에 대한 보고서를 제공받을 수 있는 무료 역량 분석 서비스

### 1.3 서비스 배경

직무 역량에 대한 관심이 높아지고 있는 반면, 기존 DISC 검사 도구들은 유료로 제공되고 있어 일반인의 접근성이 낮다. MyCPT는 LLM 기술을 활용해 DISC 검사 척도를 무료로 제공하고, 개인별 맞춤형 피드백 및 동료와의 케미 분석 기능을 제공함으로써 사용자가 자신의 역량을 파악하고 협업 과정에 활용할 수 있도록 지원한다.

### 1.4 서비스 목표

- 누구나 부담 없이 자신의 역량을 파악할 수 있도록 무료로 서비스 제공
- DISC 이론에 기반한 신뢰성 있는 분석 결과 제공
- 협업 동료 간 상호 이해 증진

## 2. 핵심 가치 및 차별점

### 2.1 핵심 가치

| 가치       | 설명                                           |
| ---------- | ---------------------------------------------- |
| **접근성** | 무료 제공, OAuth 간편 가입                     |
| **신뢰성** | DISC 이론 기반 검증된 측정 방식(강제선택) 적용 |
| **개인화** | LLM 기반 성향 맞춤 강점/약점 피드백            |
| **관계성** | 동료 케미 분석으로 협업 인사이트 제공          |

### 2.2 기술적 차별점

LLM API의 사용량에 상한을 두기 위해 **버킷 기반 캐싱 전략**을 적용하여 비용을 최소화한다.

- DISC 각 척도를 9단계로 정규화 → `9⁴ = 6,561`가지 버킷
- 동일 버킷의 결과는 DB에서 즉시 반환 (캐시 히트)
- 신규 버킷 조합에 한해 LLM API를 1회 호출하고 결과를 저장 (캐시 미스)
- 이론상 최대 **6,561회 API 호출**로 전체 사용자 커버 가능
- 1~2개월 주기 캐시 리프레시를 통해 분석 결과를 유동적으로 변화

## 3. 사용자 정의

### 3.1 비회원

- 카카오 로그인 없이 검사 응시 가능
- 검사 응시 횟수 제한 없음 (LLM 캐싱으로 비용 방어)
- 검사 완료 후 나이(출생연도), 성별 입력 필요
- 검사 완료 시 D/I/S/C 버킷값과 나이/성별을 세션에 임시 저장
- 통계 반영 및 결과 저장은 로그인 후에만 가능
- '로그인하고 결과 저장' 버튼을 통해 카카오 로그인 시 세션 데이터를 DB에 저장
- 결과 화면의 이름 표시는 '사용자'로 통일 (예: "사용자님의 강점은...")

### 3.2 회원 (카카오 로그인)

- 검사 응시 횟수 제한 없음
- 검사 결과 이력 저장 및 조회
- 결과 화면의 이름 표시는 카카오 닉네임 사용 (예: "유신님의 강점은...")
- 나이대/성별 기준 타인 통계 비교
- 동료 코드를 통한 동료 초대 및 동료 목록 관리
- 동료 케미 보고서 발행 (코인 차감)
- 코인 시스템을 통한 동료 케미 보고서 발행 횟수 관리

## 5. 비기능 요구사항

### 5.1 성능

- 캐시 히트 시 결과 응답 시간 **1초 이내**
- LLM 호출 시 결과 응답 시간 **10초 이내** (로딩 UX 제공)

### 5.2 보안

- 카카오 수집 정보는 서비스 목적 외 사용 금지
- 개인정보처리방침 동의 후 서비스 이용 가능
- DISC 분석 결과는 민감 정보로 취급하여 암호화 저장
- 동료 초대 코드는 7일 주기 갱신으로 사전 공격(코드 유추를 통한 타인 성향 열람) 방어

### 5.3 확장성

- 버킷 캐시 구조로 사용자 증가 시에도 LLM 비용 선형 증가 없음
- 통계 데이터 증가에 대비한 집계 테이블 별도 관리

### 5.4 운영

- 캐시 초기화: 배치 스케줄러로 1~2개월 주기 자동 실행
- 만료 동료 코드 삭제: 배치 스케줄러로 주기적 정리
- 오래된 읽음 알림 삭제: 배치 스케줄러로 주기적 정리 (보존 기간 정책 별도 수립)
- 서비스 면책 조항 명시: "본 결과는 참고용 성향 분석이며 심리 진단이 아닙니다"

### 6.2 주요 유즈케이스 시나리오

#### UC-01. 비회원 검사 응시 및 결과 저장

1. 사용자가 메인 페이지에서 '검사 시작' 클릭
2. 24문항 강제선택 방식으로 검사 진행
3. 검사 완료 후 출생연도 및 성별 입력
4. DISC 점수 산출 → 버킷 분류 → 캐시 조회
   - 캐시 히트: DB에서 분석 결과 즉시 반환
   - 캐시 미스: LLM API 호출 → 결과 저장 후 반환
5. 세션에 D/I/S/C 버킷값, 출생연도, 성별 임시 저장
6. 결과 화면에서 "사용자님의 강점/약점" 분석 확인
7. '로그인하고 결과 저장' 버튼 클릭 → 카카오 로그인 → 세션 데이터를 `test_results`에 INSERT → 통계 반영

#### UC-02. 회원 동료 초대 및 케미 보고서 발행

1. 회원 A가 '케미' 메뉴에서 동료 코드 확인 → 초대 링크 생성
2. A가 B에게 초대 링크 전송
3. B가 링크 접속 → 동료 등록 완료 (A↔B 양방향 관계 생성)
4. A 또는 B가 동료 목록에서 상대 선택 → '케미 보고서 발행' 클릭
5. 보유 코인 확인 후 코인 1개 차감
6. 두 사람의 최신 DISC 버킷값 기반 실시간 LLM 호출
7. 케미 보고서 저장 (발행 시점 DISC 스냅샷 포함)
8. 발행자는 보고서 확인, 상대방에게 인앱 알림 전송

## 7. 검사 설계

### 7.1 측정 이론

DISC(Dominance · Influence · Steadiness · Conscientiousness) 이론은 William Moulton Marston이 제안한 행동 성향 모델로, 4가지 차원에서 개인의 행동 패턴을 측정한다. 이론 자체는 퍼블릭 도메인이며, MyCPT는 독자적인 브랜드명과 문항으로 구성된 서비스임을 명시한다.

### 7.2 측정 방식

**강제선택(Forced Choice)** 방식을 채택한다.

- 1문항당 4개의 보기 제시 (각각 D/I/S/C 성향 대응)
- 사용자는 "나와 가장 가까운 것(Most)"과 "가장 먼 것(Least)"을 각각 1개씩 선택
- 사회적 바람직성 편향(social desirability bias)을 줄이고 상대적 성향 강도를 측정

### 7.3 문항 구성

| 척도                  | 의미                     | 문항 수       |
| --------------------- | ------------------------ | ------------- |
| D (Dominance)         | 주도형: 결단, 도전, 통제 | 24문항에 분산 |
| I (Influence)         | 사교형: 설득, 열정, 표현 | 24문항에 분산 |
| S (Steadiness)        | 안정형: 인내, 협력, 루틴 | 24문항에 분산 |
| C (Conscientiousness) | 신중형: 정확, 분석, 완벽 | 24문항에 분산 |

총 24문항, 각 문항에 D/I/S/C 보기 1개씩 포함

### 7.4 채점 방식

```
Most 선택  → 해당 척도 +2점
Least 선택 → 해당 척도 -1점
미선택     → 0점

원점수 범위: -24 ~ +48 (24문항 기준, 총 스팬 72)
검증: D + I + S + C 합계는 항상 24로 고정
```

### 7.5 점수 정규화 및 버킷 분류

원점수(-24 ~ +48, 총 스팬 72)를 9단계로 정규화한다. (72 ÷ 9 = 단계당 8점)

| 버킷 | 원점수 범위 | 의미      |
| ---- | ----------- | --------- |
| 1    | -24 ~ -17   | 최하      |
| 2    | -16 ~ -9    | 매우 낮음 |
| 3    | -8 ~ -1     | 낮음      |
| 4    | 0 ~ +7      | 보통      |
| 5    | +8 ~ +15    | 보통 이상 |
| 6    | +16 ~ +23   | 높음      |
| 7    | +24 ~ +31   | 매우 높음 |
| 8    | +32 ~ +39   | 탁월      |
| 9    | +40 ~ +48   | 최상      |

D/I/S/C 각 척도가 9단계 → 조합 수: **9⁴ = 6,561가지**

## 8. LLM 비용 최적화 전략

### 8.1 개인 분석 결과 — Lazy Caching

```
사용자 응답 완료
       ↓
D/I/S/C 원점수 산출 → 버킷 정규화 (각 1~9)
       ↓
DB SELECT WHERE d=? AND i=? AND s=? AND c=?
       ↓
캐시 HIT ──→ 즉시 반환 (LLM 호출 없음)
캐시 MISS
       ↓
Claude API 호출 (프롬프트: 버킷 조합 기반 성향 분석 요청, 이름 미포함)
       ↓
DB INSERT (d, i, s, c, analysis_result, created_at)
       ↓
사용자에게 반환 (렌더링 시 이름 삽입: "사용자님의..." / "유신님의...")
```

- 최대 **6,561회** API 호출로 모든 경우의 수 커버
- 실제 사용자 분포는 중간값 집중 → 초반부터 캐시 히트율 빠르게 상승
- 1~2개월 주기 캐시 초기화로 결과 신선도 유지
- 보고서 원문에 이름을 포함하지 않고 저장하여 LLM 프롬프트 단순화 및 비용 절감 (렌더링 시 이름 삽입)

### 8.2 동료 케미 보고서 발행 — 실시간 호출

동료 케미 보고서는 두 사람의 DISC 프로필 조합(9⁴ × 9⁴ = 약 4,300만 가지)으로 캐싱이 비효율적이며, 정확도가 서비스 신뢰도에 직결되므로 **매번 실시간 LLM 호출**을 원칙으로 한다.

- 보고서 원문 역시 이름을 포함하지 않고 저장 (렌더링 시 발행자/상대 이름 삽입)
- 사용량은 **코인 시스템**으로 통제

| 항목      | 내용                  |
| --------- | --------------------- |
| 초기 지급 | 가입 시 코인 3개      |
| 충전 주기 | 24시간마다 1개 충전   |
| 최대 보유 | 3개                   |
| 소모량    | 보고서 발행 1회당 1개 |

### 8.3 비용 절감 효과 요약

| 기능                  | 전략                        | 예상 절감 효과                   |
| --------------------- | --------------------------- | -------------------------------- |
| 개인 분석             | Lazy Caching (6,561 버킷)   | LLM 호출 약 99% 절감 (안정화 후) |
| 동료 케미 보고서 발행 | 코인 시스템으로 사용량 통제 | 무제한 호출 대비 대폭 절감       |

### 8.4 LLM 비용 정량 분석 (Claude Sonnet 4.6 기준)

#### 전제 조건

| 항목                   | 값                                     |
| ---------------------- | -------------------------------------- |
| 모델                   | Claude Sonnet 4.6                      |
| 입력 단가              | $3.00 / 1M 토큰                        |
| 출력 단가              | $15.00 / 1M 토큰                       |
| 보고서 1건당 입력 토큰 | 약 500 토큰 (시스템 프롬프트 + 버킷값) |
| 보고서 1건당 출력 토큰 | 약 1,200 토큰 (4개 섹션 본문)          |
| 전체 버킷 수           | 9⁴ = 6,561가지                         |

#### 시나리오 A — 캐시 없이 사용자마다 실시간 호출

```
호출 1회 비용
  입력:    500 ÷ 1,000,000 × $3.00  = $0.0015
  출력:  1,200 ÷ 1,000,000 × $15.00 = $0.0180
  합계:                               $0.0195 / 회 (약 28원)

사용자 10,000명 기준 누적 비용: $195 (약 28만 원)
사용자 100,000명 기준 누적 비용: $1,950 (약 280만 원)
```

#### 시나리오 B — 버킷 기반 Lazy Caching 적용

```
전체 버킷 사전 생성 시 최대 비용 (Worst Case)
  입력:  500 × 6,561 ÷ 1,000,000 × $3.00  =   $9.84
  출력: 1,200 × 6,561 ÷ 1,000,000 × $15.00 = $118.10
  합계:                                       $127.94 (약 18만 원)

캐시 초기화 주기 1~2개월 → 연간 최대 비용: 약 $768 ~ $1,536 (약 110만 ~ 220만 원)
```

#### 비용 절감 효과 비교

| 시나리오          | 사용자 10만 명 기준 비용 | 비고                               |
| ----------------- | ------------------------ | ---------------------------------- |
| 캐싱 없음         | $1,950 (약 280만 원)     | 사용자 수에 비례하여 선형 증가     |
| Lazy Caching 적용 | 최대 $128 (약 18만 원)   | 사용자 수와 무관하게 상한 고정     |
| **절감률**        | **약 93% 이상**          | 사용자가 많을수록 절감 효과 극대화 |

> 실제 Lazy Caching 환경에서는 사용자 분포가 중간값에 집중되는 경향이 있어
> 전체 6,561개 버킷 중 일부만 접근되므로, 실질 비용은 위 수치보다 더 낮게 형성된다.

## 9. 데이터 모델 개요

### 9.1 설계 원칙

- 보고서 원문(개인 분석, 케미 보고서)에는 이름을 포함하지 않고 저장. 렌더링 시 서버/클라이언트에서 이름 삽입
- 비회원 검사 결과는 세션에 임시 저장(D/I/S/C 버킷값, 출생연도, 성별)하며, 로그인 후 `test_results`에 INSERT
- 코인 잔액 변동은 `coin_transactions`에 로그로 적재하여 이상 감지 및 CS 대응 가능
- 동료 관계는 단일 행으로 저장 (양방향 조회는 UNION 쿼리로 처리)
- 케미 보고서는 발행 시점의 DISC 버킷값을 스냅샷으로 저장하여 이후 재검사로 값이 바뀌어도 보고서 내용 보존

### 9.2 주요 엔티티

```
users
  - id (PK)
  - kakao_id (UNIQUE)
  - nickname
  - birth_year
  - gender
  - coins (default: 3)
  - last_coin_charged_at
  - created_at

test_results
  - id (PK)
  - user_id (FK → users.id, nullable — 비회원은 NULL)
  - d_bucket / i_bucket / s_bucket / c_bucket  -- 버킷값 (1~9)
  - birth_year
  - gender
  - created_at

disc_cache
  - d / i / s / c  (복합 PK, 각 1~9)
  - analysis_result (TEXT)  -- 이름 미포함 보고서 원문
  - created_at

coin_transactions
  - id (PK)
  - user_id (FK → users.id)
  - amount          -- 양수: 충전, 음수: 차감
  - reason          -- ENUM (SIGNUP, RECHARGE, CHEMISTRY_REPORT)
  - balance_after   -- 트랜잭션 후 잔액 (이상 감지 용도)
  - created_at

peer_codes
  - id (PK)
  - user_id (FK → users.id, UNIQUE)
  - code (UNIQUE)   -- 고유 초대 코드 문자열
  - expires_at      -- 발급 시점 + 7일
  - created_at
  -- 만료 코드는 배치로 삭제, 코드 조회 시 만료 여부 확인 후 온디맨드 재발급

colleagues
  - id (PK)
  - user_a_id (FK → users.id)  -- 초대한 사람
  - user_b_id (FK → users.id)  -- 수락한 사람
  - created_at
  -- 양방향 조회:
  --   SELECT user_b_id FROM colleagues WHERE user_a_id = ?
  --   UNION
  --   SELECT user_a_id FROM colleagues WHERE user_b_id = ?

chemistry_reports
  - id (PK)
  - requester_id (FK → users.id)   -- 보고서 발행자
  - partner_id   (FK → users.id)   -- 보고서 대상 동료
  - report_content (TEXT)          -- 이름 미포함 보고서 원문
  -- 발행 시점 DISC 스냅샷 (이후 재검사로 값 변경되어도 보고서 내용 보존)
  - requester_d / requester_i / requester_s / requester_c
  - partner_d   / partner_i   / partner_s   / partner_c
  - created_at

notifications
  - id (PK)
  - user_id       (FK → users.id)  -- 수신자
  - type          -- ENUM (CHEMISTRY_REPORT, COLLEAGUE_REGISTERED, ...)
  - reference_id  -- 관련 엔티티 id (chemistry_reports.id 등)
  - message       -- 알림 문구
  - is_read       -- 읽음 여부 (default: false)
  - created_at
  -- 오래된 읽음 알림은 배치로 삭제 (보존 기간 정책 별도 수립)

statistics
  - age_group   -- 10대/20대/30대 등
  - gender
  - avg_d / avg_i / avg_s / avg_c
  - sample_count
  - updated_at
```
