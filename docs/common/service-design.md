## 1. 서비스 개요

### 1.1 서비스명

**MyCPT** (나의 역량, My ComPeTency)

### 1.2 서비스 개요

DISC 이론 기반의 24문항 강제선택 검사를 통해 나의 역량을 분석하고, 동료와의 협업 케미에 대한 보고서를 제공받을 수 있는 무료 역량 분석 서비스. 타인이 나의 성향을 평정하는 "나는 어떤 사람인가요?" 기능으로 자기 평정과 타인 평정을 비교할 수 있다.

### 1.3 서비스 배경

직무 역량에 대한 관심이 높아지고 있는 반면, 기존 DISC 검사 도구들은 유료로 제공되고 있어 일반인의 접근성이 낮다. MyCPT는 LLM 기술을 활용해 DISC 검사 척도를 무료로 제공하고, 개인별 맞춤형 피드백 및 동료와의 케미 분석 기능을 제공함으로써 사용자가 자신의 역량을 파악하고 협업 과정에 활용할 수 있도록 지원한다.

### 1.4 서비스 목표

- 누구나 부담 없이 자신의 역량을 파악할 수 있도록 무료로 서비스 제공
- DISC 이론에 기반한 신뢰성 있는 분석 결과 제공
- 협업 동료 간 상호 이해 증진
- 타인 평정 링크 공유를 통한 자연스러운 서비스 확산

## 2. 핵심 가치 및 차별점

### 2.1 핵심 가치

| 가치       | 설명                                                |
| ---------- | --------------------------------------------------- |
| **접근성** | 무료 제공, OAuth 간편 가입, 비회원도 검사 응시 가능 |
| **신뢰성** | DISC 이론 기반 검증된 측정 방식(강제선택) 적용      |
| **개인화** | LLM 기반 성향 맞춤 강점/약점 피드백                 |
| **관계성** | 동료 케미 분석으로 협업 인사이트 제공               |
| **다각화** | 자기 평정 + 타인 평정으로 다각적 성향 파악          |

### 2.2 기술적 차별점

LLM API의 사용량에 상한을 두기 위해 **버킷 기반 캐싱 전략**을 적용하여 비용을 최소화한다.

- DISC 각 척도를 3단계로 정규화 → `3⁴ = 81`가지 버킷
- 동일 버킷의 결과는 DB에서 즉시 반환 (캐시 히트)
- 신규 버킷 조합에 한해 LLM API를 1회 호출하고 결과를 저장 (캐시 미스)
- 이론상 최대 **81회 API 호출**로 전체 사용자 커버 가능
- 캐시 항목은 조회 시점에 만료 여부를 확인하여 온디맨드로 갱신 (접근되는 버킷만 갱신)
- 안정화 후 `disc_cache`를 Redis에 캐싱하여 DB 접근 최소화 (`@Cacheable`)

## 3. 검사 설계

### 3.1 측정 이론

DISC(Dominance · Influence · Steadiness · Conscientiousness) 이론은 William Moulton Marston이 제안한 행동 성향 모델로, 4가지 차원에서 개인의 행동 패턴을 측정한다. 이론 자체는 퍼블릭 도메인이며, MyCPT는 독자적인 브랜드명과 문항으로 구성된 서비스임을 명시한다.

### 3.2 측정 방식

**강제선택(Forced Choice)** 방식을 채택한다.

- 1문항당 4개의 보기 제시 (각각 D/I/S/C 성향 대응)
- 사용자는 "나와 가장 가까운 것(Most)"과 "가장 먼 것(Least)"을 각각 1개씩 선택
- 사회적 바람직성 편향(social desirability bias)을 줄이고 상대적 성향 강도를 측정
- DISC 태그는 서버에서 관리하며 클라이언트에 미노출 — 선택지 셔플은 클라이언트에서 처리

### 3.3 문항 구성

| 척도                  | 의미                     | 문항 수       |
| --------------------- | ------------------------ | ------------- |
| D (Dominance)         | 주도형: 결단, 도전, 통제 | 24문항에 분산 |
| I (Influence)         | 사교형: 설득, 열정, 표현 | 24문항에 분산 |
| S (Steadiness)        | 안정형: 인내, 협력, 루틴 | 24문항에 분산 |
| C (Conscientiousness) | 신중형: 정확, 분석, 완벽 | 24문항에 분산 |

총 24문항, 각 문항에 D/I/S/C 보기 1개씩 포함

### 3.4 채점 방식

```
Most 선택  → 해당 척도 +2점
Least 선택 → 해당 척도 -1점
미선택     → 0점

원점수 범위: -24 ~ +48 (24문항 기준, 총 스팬 72)
검증: D + I + S + C 합계는 항상 24로 고정
```

### 3.5 점수 정규화 및 버킷 분류

원점수(-24 ~ +48, 총 스팬 72)를 통계적 정규분포를 반영한 불균등 3구간으로 정규화한다.

| 버킷 | 원점수 범위 | 의미                                                |
| ---- | ----------- | --------------------------------------------------- |
| 1    | -24 ~ -5    | Low — 해당 성향을 의도적으로 기피/거부              |
| 2    | -4 ~ +11    | Mid — 데이터 밀집 중심부. 상황에 따라 유연하게 발현 |
| 3    | +12 ~ +48   | High — 데이터 희소 영역. 확실한 주성향              |

구간 폭이 불균등한 이유:

- Mid(-4 ~ +11)는 강제선택형 검사 특성상 유저 데이터가 가장 밀집되는 구간
- High(+12 ~ +48)는 실제로 도달하기 어려운 희소 구간으로 확실한 주성향을 의미
- 균등 분할(9구간) 대비 캐시 히트율 극대화 (6,561 → 81가지)

D/I/S/C 각 척도가 3단계 → 조합 수: **3⁴ = 81가지**

## 4. LLM 비용 최적화 전략

### 4.1 개인 분석 결과 — Lazy Caching + 온디맨드 만료

```
사용자 응답 완료
↓
D/I/S/C 원점수 산출 → 버킷 정규화 (각 1~3)
↓
DB SELECT WHERE d=? AND i=? AND s=? AND c=?
↓
캐시 미스 → LLM 호출 → INSERT → 반환
캐시 히트 → created_at 확인
↓
유효 (created_at >= 기준) → 즉시 반환
만료 (created_at <  기준) → LLM 호출 → UPDATE → 반환
```

- 최대 **81회** API 호출로 모든 경우의 수 커버
- 실제 접근되는 버킷만 갱신 → 전체 초기화 대비 비용 최소화
- 만료 기준은 애플리케이션 설정값으로 외부화 (`cache.ttl-days=365` 등)
- 보고서 원문에 이름을 포함하지 않고 저장 (렌더링 시 이름 삽입)

### 4.2 동료 케미 보고서 발행 — 비동기 실시간 호출

동료 케미 보고서는 두 사람의 DISC 프로필 조합(3⁴ × 3⁴ = 6,561가지)으로 캐싱이 비효율적이며, 정확도가 서비스 신뢰도에 직결되므로 **매번 실시간 LLM 호출**을 원칙으로 한다.

- `@Async`로 비동기 처리 → 클라이언트 대기 시간 없음
- 발행 완료 시 SSE로 발행자에게 실시간 푸시
- 보고서를 Markdown 단일 TEXT로 저장 — 검사 유형(DISC/MBTI/BIG5) 추가 시 스키마 변경 없이 프롬프트만 수정
- 보고서 원문에 이름 미포함 저장 (렌더링 시 발행자/상대 이름 삽입)
- DISC 버킷 스냅샷 미저장 — 보고서 텍스트가 성향 정보를 충분히 포함
- 사용량은 **코인 시스템**으로 통제

| 항목      | 내용                  |
| --------- | --------------------- |
| 초기 지급 | 가입 시 코인 3개      |
| 충전 주기 | 24시간마다 1개 충전   |
| 최대 보유 | 3개                   |
| 소모량    | 보고서 발행 1회당 1개 |

코인 충전은 온디맨드 방식으로 처리한다. 코인 관련 요청 시 `next_coin_at <= NOW()`를 확인하여 그 자리에서 충전량을 계산 후 반영한다.

```
elapsed    = (NOW() - next_coin_at) / 24시간
chargeable = Math.floor(elapsed) + 1
amount     = Math.min(chargeable, 3 - current_coins)
```

### 4.3 비용 절감 효과 요약

| 기능                  | 전략                        | 예상 절감 효과                     |
| --------------------- | --------------------------- | ---------------------------------- |
| 개인 분석             | Lazy Caching (81 버킷)      | LLM 호출 약 99.9% 절감 (안정화 후) |
| 동료 케미 보고서 발행 | 코인 시스템으로 사용량 통제 | 무제한 호출 대비 대폭 절감         |

### 4.4 LLM 비용 정량 분석 (Claude Sonnet 4.6 기준)

#### 전제 조건

| 항목                   | 값                                     |
| ---------------------- | -------------------------------------- |
| 모델                   | Claude Sonnet 4.6                      |
| 입력 단가              | $3.00 / 1M 토큰                        |
| 출력 단가              | $15.00 / 1M 토큰                       |
| 보고서 1건당 입력 토큰 | 약 500 토큰 (시스템 프롬프트 + 버킷값) |
| 보고서 1건당 출력 토큰 | 약 2,000 토큰 (Markdown 보고서 전문)   |
| 전체 버킷 수           | 3⁴ = 81가지                            |

#### 시나리오 A — 캐시 없이 사용자마다 실시간 호출

```
호출 1회 비용
입력:    500 ÷ 1,000,000 × $3.00  = $0.0015
출력:  2,000 ÷ 1,000,000 × $15.00 = $0.0300
합계:                               $0.0315 / 회 (약 46원)

사용자 10,000명 기준 누적 비용: $315 (약 46만 원)
사용자 100,000명 기준 누적 비용: $3,150 (약 460만 원)
```

#### 시나리오 B — 버킷 기반 Lazy Caching 적용

```
전체 버킷 사전 생성 시 최대 비용 (Worst Case)
입력:   500 × 81 ÷ 1,000,000 × $3.00  = $0.12
출력: 2,000 × 81 ÷ 1,000,000 × $15.00 = $2.43
합계:                                     $2.55 (약 3,700원)
```

#### 비용 절감 효과 비교

#### 비용 절감 효과 비교

| 시나리오          | 사용자 10만 명 기준 비용 | 비고                               |
| ----------------- | ------------------------ | ---------------------------------- |
| 캐싱 없음         | $3,150 (약 460만 원)     | 사용자 수에 비례하여 선형 증가     |
| Lazy Caching 적용 | 최대 $2.55 (약 3,700원)  | 사용자 수와 무관하게 상한 고정     |
| **절감률**        | **약 99.9% 이상**        | 사용자가 많을수록 절감 효과 극대화 |
